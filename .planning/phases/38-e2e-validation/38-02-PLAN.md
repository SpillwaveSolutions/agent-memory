**Test 1: `test_stale_results_downranked_relative_to_newer`**
- Create 4 separate sessions, each with events at different timestamp offsets (0d, 14d, 28d, 42d ago from base_ts). All events discuss the same topic (e.g., "Rust memory management and borrow checker patterns") so BM25 finds them all for the same query.
- For each session: create events using `create_test_events` with modified timestamps, ingest via `ingest_events()`, build TOC via `build_toc_segment()`, index into BM25 via `SearchIndexer`.
- Create `RetrievalHandler::with_services(storage, Some(bm25_searcher), None, None, StalenessConfig { enabled: true, half_life_days: 14.0, max_penalty: 0.30, ..Default::default() })`.
- Call `handler.route_query()` with query matching the topic.
- Assert relative ordering: results from newer sessions score >= results from older sessions. Use `assert!(newer_score >= older_score, "...")` for each adjacent pair. Do NOT assert exact percentages (per user decision).

**Test 2: `test_kind_exemption_constraint_not_penalized`**
- Create two sessions at different timestamps (0d and 42d ago).
- The 42d-old session contains events with `memory_kind: "constraint"` in metadata (high-salience, exempt from decay per RETRV-03).
- The 0d session contains regular observation events.
- Ingest, TOC, index, route_query with StalenessConfig enabled.
- Assert: the old Constraint events are NOT heavily penalized -- their scores should be comparable to (or not drastically lower than) the newest results. The key assertion is that Constraint kind exemption is working. Compare old-constraint scores vs what a regular old-observation would get. Since we can't test exact percentages, verify that the constraint event appears in results (i.e., was not dropped to bottom) despite being 42 days old.

Note: If `memory_kind` metadata doesn't propagate through BM25 indexing to `SearchResult.metadata`, test StaleFilter directly with hand-crafted `SearchResult` objects (per RESEARCH.md recommendation). Create a simpler test that calls `StaleFilter::apply_with_supersession()` directly with results containing `memory_kind: "constraint"` vs `memory_kind: "observation"` and asserts the constraint result retains its original score.

**Test 3: `test_stale_filter_disabled_no_score_change`**
- Same setup as Test 1 but with `StalenessConfig { enabled: false, ..Default::default() }`.
- Route query and verify that results are returned (has_results=true) without staleness reranking applied. This is a control test proving the filter is opt-in.
